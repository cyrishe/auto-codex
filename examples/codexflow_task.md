# CodexFlow Issue Template

> 使用方式：把本模板复制到 GitHub issue。请删除不相关段落，但不要删除“目标、非目标、验收标准、评估方式、约束”这几个核心段落。

## 标题

`[模块/场景] 用一句话说明要交付的行为`

示例：

`[portfolio] Add daily PnL summary endpoint`

## 背景 / 当前行为

说明为什么要做这个任务，以及当前行为是什么。

建议包含：

- 当前用户或系统遇到的问题。
- 当前代码已有的相关能力。
- 为什么现在需要改。

## 目标

明确本 issue 必须完成什么。

请使用可执行、可验证的描述：

- 实现什么用户可见行为。
- 修改或新增哪些接口、命令、页面、任务或数据流。
- 需要兼容哪些已有行为。

## 非目标

明确本次不做什么，避免扩大范围。

示例：

- 不重构无关模块。
- 不改变已有公开接口，除非“目标”中明确说明。
- 不引入新的外部服务。

- 兼容旧数据：

## 验收标准

所有勾选项都必须满足，才算本 issue 完成。

- [ ] 核心行为符合“目标”和“需求细节”。
- [ ] “边界情况”中列出的场景有处理。
- [ ] 已添加或更新必要测试。
- [ ] 已通过“评估方式”中的命令。
- [ ] 没有扩大到“非目标”范围。
- [ ] 没有修改敏感文件、生产配置或无关模块。

## 可能涉及的模块

- `src/...`
- `tests/...`

如果不确定，可以写“可能”，不要强行指定实现方式。

## 评估方式

开发完成后必须运行以下命令。命令应尽量 deterministic，不能依赖本地私有状态。

```bash
pytest -q
```

如果需要额外人工验证，请写清楚步骤和预期结果：

1. 启动：
2. 操作：
3. 预期：

## 约束

- 不要破坏旧接口。
- 不要引入新依赖，除非本 issue 明确要求。
- 不要修改生产配置或敏感文件。
- 保持改动聚焦在本 issue。
- 如果发现需求不清或实现风险过大，应停止并标记 blocked，而不是自行扩大范围。

## 推荐 Label

- `codex:ready`：可以开始执行。
- `codex:working`：CodexFlow 已领取并处理中。
- `codex:review`：本地开发、测试和 review 已完成，等待用户审查。
- `codex:blocked`：需求、环境、测试或权限阻塞，需要用户处理。
